### Missed Call Pattern Categories

**Category 1: External Framework Methods (confirmed root cause — highest volume)**
- **Definition:** Callee methods belong to external libraries (ASP.NET, Spring Boot, MediatR, Playwright, AutoMapper) that are NOT indexed by Synapps (not in the filesystem being indexed).
- **C# examples:** `sender.Send()` (MediatR), `builder.Services.AddScoped()` (DI), `TypedResults.NoContent()` (ASP.NET), `Assertions.Expect()` (Playwright)
- **Java examples:** `SpringApplication.run()`, `restClient.get()`, `vetRepository.findAll()` (Spring Data JPA), `vectorStore.similaritySearch()` (Spring AI)
- **Why missed:** `SymbolResolver` resolves callee names against the graph. External framework types have no Method nodes in the graph, so `MATCH (dst:Method {full_name: $callee})` returns nothing → edge not written.
- **Impact:** Very high. In annotation-heavy frameworks like Spring Boot and ASP.NET minimal APIs, most business logic delegates to framework interfaces. Real-world projects will show 60-80% of methods with zero outgoing CALLS.
- **Fixability:** Intentional by design — Synapps indexes project code, not external dependencies. The CALLS graph correctly represents intra-project calls.

**Category 2: POJO Getters/Setters and Pure Assignments (expected — no fix needed)**
- **Definition:** Trivial methods that only read or write a field with no method calls.
- **Java examples:** `Vet.getId()`, `Vet.setFirstName()`, `Pet.getBirthDate()`, `PetType.getName()`
- **C# examples:** Auto-property accessors (C# indexer sees these as Method nodes in some cases)
- **Impact:** Medium volume (Java: ~51 of 132 zero-outgoing methods are getters/setters). These are genuinely call-free methods.
- **Fixability:** None needed — zero outgoing CALLS is accurate.

**Category 3: Fluent API / Method Chaining (missed)**
- **Definition:** Methods whose body is a single fluent chain expression (`restClient.get().uri(…).retrieve().body(…)`). The individual chained calls resolve to external library objects.
- **Java examples:** `AIDataProvider.getAllOwners()` chains 4 calls on `RestClient` (external); `PetclinicChatClient.exchange()` chains 6 calls on `ChatClient` (Spring AI)
- **C# examples:** `WeatherPage.AssertHeading()` is an expression-body with chained Playwright calls
- **Why missed:** Even when Synapps detects the outer call site, the resolver cannot match intermediate calls in a chain to indexed methods because each intermediate return type is from an external library.
- **Fixability:** Low priority. Fluent chain resolution would require type inference on intermediate return types — this is LSP-level work far beyond simple name matching.

**Category 4: Constructor Body Assignments (expected)**
- **Definition:** Constructors that only perform `this.field = param` assignments.
- **Java examples:** `PetclinicTools(AIDataProvider)`, `VetResource(VetRepository)`, `AIDataProvider(VectorStore, DiscoveryClient)`
- **C# examples:** Primary constructors (e.g., `WeatherPage(IPage page)`) — the constructor IS the field assignment.
- **Fixability:** None needed — field assignments are not method calls.

**Category 5: Spring Data JPA Interface Methods (missed — medium priority)**
- **Definition:** Java `@Repository` interfaces (`JpaRepository`, `CrudRepository`) have methods like `findAll()`, `save()`, `findById()` that callers invoke but are not indexed as Method nodes because they are interface default implementations from an external JAR.
- **Java examples:** `vetRepository.findAll()`, `ownerRepository.save()`, `ownerRepository.findById()`
- **Why missed:** The JPA repository is declared as an interface in project code (`VetRepository extends JpaRepository<Vet, Integer>`), but `findAll()` is inherited from `JpaRepository` in an external JAR — not in the project's source tree. The interface IS indexed as a Class node, but the inherited methods are not.
- **Fixability:** Medium effort — could detect calls to methods on `@Repository`-annotated types and create CALLS edges to the interface method even if unresolved. This would require recognizing Spring Data method naming conventions.

**Category 6: Expression-Body Members / Lambda Calls (missed)**
- **Definition:** C# expression-body methods (`=> SomeCall(...)`) and Java lambda methods. The call extractor currently handles method invocations but some expression body patterns may be parsed differently by tree-sitter.
- **C# examples:** `WeatherPage.AssertHeading(string text) => Assertions.Expect(…).ToHaveTextAsync(text)`
- **Impact:** Low (only 3 Playwright page object methods in sample)
- **Fixability:** May be a tree-sitter parsing gap. The `_CSHARP_CALLS_QUERY` should handle expression bodies if the tree-sitter C# grammar exposes them as `invocation_expression` nodes. Needs targeted investigation.

### Recommendations

**Recommendation 1 (Accept): External framework calls not appearing is expected behavior**
The ~60-78% zero-outgoing rate in these projects is not a bug — it accurately reflects that MediatR, ASP.NET, Spring Boot, and Spring AI framework calls are not indexed (they're in external JARs or NuGet packages). This is the correct design: Synapps indexes project-level code.

**Recommendation 2 (Document): Surface the limitation to users**
When `find_dead_code` or `find_untested` returns methods that appear to do nothing, users should understand that "zero outgoing CALLS" can mean either (a) the method genuinely does nothing or (b) it delegates only to external framework code. Consider adding a note to the `find_dead_code` MCP tool documentation.

**Recommendation 3 (Investigate — Future Phase): Expression-body C# calls**
The 3 Playwright expression-body methods in the sample should have detectable call sites. A targeted tree-sitter investigation to confirm whether `invocation_expression` inside `=> expr` bodies is covered is worthwhile. If it's a gap, the fix is a 1-line query change.

**Recommendation 4 (Reject): Spring Data JPA interface method resolution**
Creating synthetic CALLS edges to JPA interface methods would require heuristic detection of Spring Data conventions (`findBy*`, `save`, `delete`) — fragile and domain-specific. Deferred.

## C# IMPORTS

### Stats

| Metric | Value |
|--------|-------|
| C# files in CleanArchitecture | 110 |
| IMPORTS edges (CleanArchitecture) | 24 |
| Distinct Package nodes referenced | 10 |
| Total C# files across all indexed projects | 396 |
| Total IMPORTS edges (all C# projects) | 375 |

### Analysis

**The original concern (from RESEARCH.md section 4) was 29 IMPORTS edges for 153 files. The current data shows 375 edges for 396 files — substantially more than the concern anticipated. The discrepancy in the reported 29 figure likely came from querying against a different indexed snapshot or different project.**

For CleanArchitecture specifically (110 C# files, 24 IMPORTS edges), the IMPORTS edge targets are:

```
CleanArchitecture.Application.Common.Interfaces   (internal — present)
CleanArchitecture.Application.Common.Models        (internal — present)
CleanArchitecture.Application.Common.Security      (internal — present)
CleanArchitecture.Infrastructure.Identity          (internal — present)
CleanArchitecture.Shared                           (internal — present)
Microsoft.Extensions.DependencyInjection           (external — no Package node but IMPORTS edge exists!)
Microsoft.Extensions.Hosting                       (external — same)
src/Web/ClientApp-React/src/components/...         (TypeScript path — misclassified)
src/Web/ClientApp/src/api-authorization/...        (TypeScript path — misclassified)
```

**Critical finding:** IMPORTS edges ARE being created for external namespaces (`Microsoft.Extensions.*`). This contradicts the RESEARCH.md hypothesis that "IMPORTS edges only form when the namespace Package node exists in the graph." Investigation reveals:

1. **Package nodes exist without a `language` field** — all 87 Package nodes in the graph have `language = null`. The earlier query filtered by `language = 'csharp'` which returned 0. Package nodes are language-agnostic in the schema.

2. **`upsert_imports` uses MERGE-on-create semantics** — the function creates Package nodes on-the-fly when writing IMPORTS edges. The code in `graph/nodes.py` creates Package nodes as needed rather than requiring them to pre-exist. This means the "only 29 edges" concern was based on a misunderstanding of the code.

3. **TypeScript paths appearing as IMPORTS targets** — three IMPORTS edge targets have file paths like `src/Web/ClientApp/src/api-authorization/auth.service.AuthService`. This appears to be a cross-language artifact where C# `using` directives that reference generated TypeScript proxy types are creating IMPORTS edges to non-existent Package nodes with TypeScript path strings. This is a minor data quality issue.

4. **Why only 24 edges for 110 files?** — Most C# `using` directives reference external NuGet packages (e.g., `System`, `Microsoft.AspNetCore.*`) that the C# LSP identifies as external namespace types. The `CSharpImportExtractor` returns all `using` directive names, but the upstream LSP adapter or indexer may be filtering. Investigation of the `_index_file_imports` path shows that `upsert_imports` does a MERGE on Package nodes — so ALL extracted using directives should produce Package nodes. The 24 edges vs 110 files gap means most files have 0 or 1 internal IMPORTS edges, which matches CleanArchitecture's clean architecture design where layers have minimal cross-references.

### Findings Summary

The C# IMPORTS edge count is **not a bug**. The low ratio (24 edges for 110 files = 0.22 per file) reflects CleanArchitecture's intentional layering: most files only `using` external framework namespaces (System, Microsoft.*, AutoMapper, MediatR), and those do create Package nodes but the same Package node is reused across files (MERGE semantics).

**The TypeScript path misclassification** (`src/Web/ClientApp/...` appearing as Package targets for C# files) is a minor data quality issue worth investigating in a future phase. It suggests the C# import extractor is parsing some non-`using` construct, or the LSP is returning unexpected references.

### Recommendations

**Recommendation 1 (Accept): No fix needed for IMPORTS edge count**
The C# IMPORTS count is behaving correctly. Package nodes are created on demand, edges form per MERGE. Low ratios are expected for frameworks like CleanArchitecture where most cross-cutting concerns come from NuGet packages that resolve to the same handful of internal namespace Package nodes.

**Recommendation 2 (Investigate — Future Phase): TypeScript path IMPORTS targets**
The 2-3 CleanArchitecture IMPORTS edges pointing to TypeScript file paths (e.g., `src/Web/ClientApp/src/api-authorization/auth.service.AuthService`) indicate a parsing artifact. Investigate whether the C# extractor is processing `using` aliases that reference TypeScript generated proxy types, or whether there is a source of cross-language edge creation. Low severity (2-3 edges), but surprising.

**Recommendation 3 (No action): Package nodes lack `language` field**
Package nodes are intentionally language-agnostic (a namespace like `CleanArchitecture.Application.Common.Interfaces` is shared by C# files across multiple layers). The `language = null` is by design. Queries that filter by `language` on Package nodes should not expect a language value.

## Cross-Cutting Recommendation

Both investigations confirm that Synapps' CALLS and IMPORTS coverage is **working as designed**. The apparent gaps come from:

1. External library calls (expected — not indexed)
2. Framework-specific patterns (Spring DI, Spring Data JPA) that require runtime injection
3. Trivial methods (getters/setters, empty constructors) that genuinely have no calls

No code fixes are warranted from this investigation. The CALLS coverage of 22-40% for framework-heavy projects is the realistic baseline for static analysis without external dependency indexing.

**Priority for future phase:**
- LOW: Expression-body C# call detection (potential tree-sitter gap)
- LOW: TypeScript path IMPORTS artifact investigation
- REJECT: Spring Data JPA synthetic edges, external framework coverage
